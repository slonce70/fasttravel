# FastTravel — operations runbook

Single source of truth for day-2 operations: secrets, restores,
backups, monitoring tweaks. Read this BEFORE `git push` to prod for
the first time.

The audit-closure work (May 2026) added a handful of new required
secrets and renamed a few defaults — if your `.env` predates 2026-05-27,
read the **"What changed"** sections carefully.

---

## 1. Secret inventory

Everything in here is generated by you (or by `infra/scripts/secrets-bootstrap.sh`)
and put into either the VPS `.env`, GitHub Actions secrets, or both.

### VPS `.env` (lives at `/opt/fasttravel/.env`, mode 600, owned by `fasttravel`)

| Var | Required | How to get it |
|---|---|---|
| `ENVIRONMENT=prod` | yes | literal |
| `POSTGRES_USER` / `POSTGRES_PASSWORD` / `POSTGRES_DB` | yes | `secrets-bootstrap.sh` generates a `hex 32` password |
| `DATABASE_URL` / `DATABASE_URL_SYNC` | yes | composite of the above |
| `REDIS_URL` | yes | `redis://redis:6379/0` (no auth in compose — internal bridge only) |
| `TELEGRAM_BOT_TOKEN` | yes | @BotFather — **rotate to a NEW token; never reuse the dev one** (audit P0 #1) |
| `TELEGRAM_CHANNEL_ID` | yes | `-100…` numeric id of the prod channel |
| `ALERTMANAGER_WEBHOOK_SECRET` | yes | `openssl rand -hex 32` — also put in `infra/prometheus/.webhook_secret` (gitignored) |
| `GRAFANA_ADMIN_PASSWORD` | yes | `openssl rand -hex 32` — compose `${GRAFANA_ADMIN_PASSWORD:?…}` refuses to boot without it |
| `API_IMAGE` / `BOT_IMAGE` / `SCHEDULER_IMAGE` | yes | set by deploy workflow at runtime to `ghcr.io/<owner>/<repo>-{api,bot,scheduler}:sha-XYZ` |
| `SENTRY_DSN` | optional | from sentry.io project settings; empty = Sentry disabled |
| `PUBLIC_SITE_URL` | bot | `https://fasttravel.com.ua` so inline buttons can build `/hotels/{slug}` URLs |

### GitHub Actions secrets (Settings → Secrets and variables → Actions)

| Secret | Required by | Notes |
|---|---|---|
| `DEPLOY_SSH_HOST` | deploy-api | IP of the VPS |
| `DEPLOY_SSH_USER` | deploy-api | `fasttravel` |
| `DEPLOY_SSH_KEY` | deploy-api | private key (OpenSSH PEM) |
| `DEPLOY_SSH_KNOWN_HOSTS` | deploy-api | **REQUIRED** since the audit — TOFU fallback removed. Capture once: `ssh-keyscan -H <vps_ip>` |
| `BACKUP_SSH_HOST` / `USER` / `KEY` / `KNOWN_HOSTS` | daily-backup | same shape as the deploy set; usually the same values |
| `R2_ACCOUNT_ID` / `R2_ACCESS_KEY_ID` / `R2_SECRET_ACCESS_KEY` / `R2_BUCKET` | daily-backup | Cloudflare R2 token + bucket name |
| `BACKUP_AGE_RECIPIENT` | daily-backup | **REQUIRED** since the audit — age public key (`age1…`); see "Backups" below |
| `CLOUDFLARE_ACCOUNT_ID` / `CLOUDFLARE_API_TOKEN` | deploy-web | for `wrangler` Worker deploy |
| `CODECOV_TOKEN` | ci (optional) | coverage upload; CI won't fail without it |
| `DEPLOY_NOTIFY_WEBHOOK` | deploy-api (optional) | Slack/Discord webhook on deploy failure |

## 2. First-time setup checklist

After you have the VPS and the Cloudflare R2 bucket:

```bash
# 1. Generate age keypair (locally, ONCE)
brew install age   # or apt-get install age
age-keygen -o ~/.config/fasttravel/backup.key
# → prints "Public key: age1..."  ← save this to GitHub secret BACKUP_AGE_RECIPIENT

# Move private key out of disk:
#   • password manager entry "FastTravel age backup key"
#   • OR YubiKey age-plugin
#   • OR printed + safe deposit box
# The whole point is: backups become useless without the private key.

# 2. Generate webhook secret + grafana password
openssl rand -hex 32   # → ALERTMANAGER_WEBHOOK_SECRET
openssl rand -hex 32   # → GRAFANA_ADMIN_PASSWORD

# 3. Capture VPS host key
ssh-keyscan -H <vps_ip> | base64
# → paste base64-decoded result into BACKUP_SSH_KNOWN_HOSTS + DEPLOY_SSH_KNOWN_HOSTS

# 4. Generate prod .env on the VPS
ssh fasttravel@<vps_ip>
cd /opt/fasttravel
./infra/scripts/secrets-bootstrap.sh .env
$EDITOR .env       # paste:
                   #   TELEGRAM_BOT_TOKEN (NEW token, not the dev one)
                   #   TELEGRAM_CHANNEL_ID
                   #   ALERTMANAGER_WEBHOOK_SECRET (same as GH secret)
                   #   GRAFANA_ADMIN_PASSWORD
                   #   SENTRY_DSN (optional)
                   #   PUBLIC_SITE_URL
chmod 600 .env

# 5. Mirror the webhook secret into AlertManager
echo "$ALERTMANAGER_WEBHOOK_SECRET" > infra/prometheus/.webhook_secret
chmod 600 infra/prometheus/.webhook_secret
# (also gitignored — never committed)

# 6. Run preflight
ENV_FILE=.env STRICT_ENV=1 ./infra/scripts/production-preflight.sh

# 7. First deploy
docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d
```

## 3. Backups & restore

### How it works

`.github/workflows/daily-backup.yml` runs daily 01:00 UTC:

1. SSH into VPS, runs `pg_dump -Fc --compress=9` inside the Postgres
   container → `/tmp/fasttravel-backup-<stamp>.dump`.
2. `scp` to the GitHub Actions runner.
3. `pg_restore --list` validates the file structurally (PGDMP magic).
4. Spins up a temp Postgres container, runs `pg_restore -j 4` against it
   (restore drill — proves the dump is restorable, not just well-formed).
5. **Encrypts** with age (`age -r $BACKUP_AGE_RECIPIENT`) → ciphertext only.
6. `rclone copy` to R2 bucket `postgres/` prefix.
7. Retains the most recent 30 `.age` files; deletes older.

### Restoring a backup

You'll need:
- The private age key (the file you should have moved to your password
  manager during first-time setup).
- An empty Postgres instance to restore into (probably a fresh local
  container).

```bash
# 1. Download the encrypted dump from R2
rclone copy r2:fasttravel-backups/postgres/<stamp>.dump.age ./

# 2. Decrypt
age -d -i ~/.config/fasttravel/backup.key \
    -o restored.dump <stamp>.dump.age

# 3. Restore (parallel)
pg_restore -j 4 --no-owner --clean --if-exists \
    -U <user> -d <db> restored.dump

# 4. Verify
psql -U <user> -d <db> -c "
  SELECT count(*) AS hotels FROM hotels;
  SELECT count(*) AS deals FROM deals WHERE detected_at > NOW() - INTERVAL '7 days';"
```

### WAL archiving (off by default)

The `pg_dump` cycle leaves a ~24h RPO. To get near-zero RPO, enable
WAL archiving to R2:

1. Install `rclone` on the VPS host (not container) with the same `[r2]`
   config the backup workflow uses.
2. Copy `infra/postgres/pg_wal_archive.sh` to `/usr/local/bin/` (chmod 755).
3. Edit `infra/postgres/postgresql.conf`:
   ```
   archive_mode    = on
   archive_command = '/usr/local/bin/pg_wal_archive.sh "%p" "%f"'
   ```
4. `docker compose restart postgres`
5. Verify: `docker compose exec postgres psql -U fasttravel -c
   'SELECT * FROM pg_stat_archiver;'` — `archived_count` should climb,
   `failed_count` stays at 0.

Recovery uses `restore_command` pulling from R2 + the latest base dump.

## 4. Mypy ratchet — refreshing the baseline

CI enforces "mypy errors per service must not exceed baseline".
Baselines live in `infra/scripts/mypy-baseline.txt`.

After a typing cleanup (you reduced the count):

```bash
./infra/scripts/mypy-ratchet.sh --refresh-baseline api scheduler bot
git add infra/scripts/mypy-baseline.txt
git commit -m "chore(mypy): refresh ratchet baseline after typing cleanup"
```

Current baseline (May 2026): `api=0`, `scheduler=0`, `bot=70`.
The bot's 70 are pre-existing aiogram `Message | InaccessibleMessage |
None` union-narrowing gaps; ratchet just prevents regression while a
targeted PR closes them.

## 5. Observability

### Default: OFF (opt-in)

To keep the $0 single-VM deploy lean, `docker compose up` runs only
postgres/redis/api/bot/scheduler. The entire monitoring stack —
Prometheus, AlertManager, Grafana, node/postgres/redis exporters, and
Loki + Vector — sits behind the `observability` profile:

```bash
docker compose --profile observability up -d
# adds: prometheus, alertmanager, grafana, node/postgres/redis exporters,
#       loki (port 3100 internal), vector (tails containers → loki)
```

- 9 alert rules in `infra/prometheus/rules/fasttravel.yml`. AlertManager
  posts firing alerts to the bot's `/alerts` endpoint → Telegram **when the
  bot has `ALERT_WEBHOOK_ENABLED=true`** (off by default; turn it on
  together with the profile + `ALERTMANAGER_WEBHOOK_SECRET`).
- node + postgres + redis exporters scraped on the `fasttravel` network.
- Add Loki as a Grafana data source (`http://loki:3100`).

Check `docker stats` before enabling on a free-tier VM — the full stack
adds a few hundred MB RSS (Loki alone ~150-300 MB).

## 6. Image rollback

After every deploy, `api-<sha>` / `bot-<sha>` / `scheduler-<sha>` tags
live in GHCR. The `:latest` tag is intentionally **not pushed** (audit
4.2 — it defeated sha-pin rollback).

To roll back:

```bash
# On the VPS, find a previous sha
docker images | grep ghcr.io/<owner>/<repo>-api

# Re-run compose with the rollback sha
export API_IMAGE=ghcr.io/<owner>/<repo>-api:sha-PREVIOUS
export BOT_IMAGE=ghcr.io/<owner>/<repo>-bot:sha-PREVIOUS
export SCHEDULER_IMAGE=ghcr.io/<owner>/<repo>-scheduler:sha-PREVIOUS
docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d \
    --no-deps api bot scheduler
```

## 7. Secret rotation

Every secret listed in §1 should be rotated periodically. Suggested cadence:

| Secret | Cadence | How |
|---|---|---|
| `TELEGRAM_BOT_TOKEN` | annually or on suspected compromise | @BotFather → Revoke + new token; update VPS `.env` and `docker compose restart bot` |
| `ALERTMANAGER_WEBHOOK_SECRET` | annually | `openssl rand -hex 32`; update `.env` + `.webhook_secret`; `docker compose restart bot alertmanager` |
| `GRAFANA_ADMIN_PASSWORD` | annually | same; `docker compose restart grafana` |
| `POSTGRES_PASSWORD` | annually | rotate user via `ALTER USER`; update `DATABASE_URL` in `.env`; rolling restart |
| `BACKUP_AGE_RECIPIENT` | only on compromise | new keypair; re-encrypt existing R2 dumps OR accept old dumps stay readable only with old key |
| `R2_ACCESS_KEY_ID` / `_SECRET` | annually | mint new token in Cloudflare → update GH secret |
| SSH key | annually | new keypair; add new public key to `authorized_keys` BEFORE removing old |

The bot's old token (the one that was in the dev `.env` when the audit
ran) is considered compromised per audit P0 #1 and **must** be rotated
before first prod use.

## 8. Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| `RefuseToStart: Refusing to start in prod with unsafe…` | `.env` still has `_change_me` or missing required secret | check `Settings.assert_prod_secrets()` offenders list in the log |
| AlertManager → bot returns 503 | `ALERTMANAGER_WEBHOOK_SECRET` empty in prod | set the secret in `.env`, restart bot |
| Bot starts but `/best` returns "сервіс недоступний" | api service down or `API_BASE_URL` wrong | `docker compose ps api`, check API logs |
| Deploy workflow gated as "queued" forever | CI failed on same SHA — `workflow_run` waits for `conclusion: success` | check CI run; fix red, push, deploy fires automatically |
| `BACKUP_AGE_RECIPIENT secret(s) not configured` | missing in GH secrets | set it (the age public key from §2) |
| `DEPLOY_SSH_KNOWN_HOSTS secret is required` | TOFU fallback removed | capture via `ssh-keyscan -H <vps>`, add to GH secret |
| Grafana boot loop | `GRAFANA_ADMIN_PASSWORD` unset | set in `.env`; restart |
| `pg_stat_archiver.failed_count` climbing | rclone misconfigured or R2 token revoked | check `/var/log/postgresql/*` for the archive_command stderr; fix rclone, restart postgres |

## 9. Runtime tuning knobs

These values are optional. Leave them at the shipped defaults unless logs,
metrics, or a recovery run show a concrete need to change them.

| Var | Default | Service | When to change it |
|---|---:|---|---|
| `RATE_LIMIT_TRUSTED_HEADER` | empty = `CF-Connecting-IP,X-Forwarded-For` | api | Only when a trusted proxy chain other than Cloudflare/nginx sits in front of the API. Keep the first spoof-protected client-IP header first. |
| `FT_FARVATER_CONCURRENCY` | `3` | scheduler | Snapshot job fan-out. In that job, this also becomes the HTTP-client concurrency fallback when `FT_FARVATER_HTTP_CONCURRENCY` is unset. |
| `FT_FARVATER_HTTP_CONCURRENCY` | `3` | scheduler | Farvater HTTP-client concurrency. Set it explicitly when it should differ from snapshot fan-out; raising it is the fastest way to trigger 429/403 storms. |
| `FT_FARVATER_DAILY_CAP` | `0` | scheduler | Daily Farvater request cap. `0` disables enforcement; set a cap during recovery or catalog expansion when you need a hard upstream budget. |
| `FT_FARVATER_HTTP_TIMEOUT_S` | `30.0` | scheduler | Per-request Farvater HTTP timeout. Increase only for confirmed upstream slowness; lower only after checking retry/error logs. |
| `FT_SITEMAP_INGEST_CONCURRENCY` | `12` | scheduler | Long-tail sitemap ingest fan-out. Lower before enabling startup ingest on a small VM or after 429/timeout evidence. |
| `FT_SITEMAP_INGEST_DELAY_S` | `0.05` | scheduler | Per-request delay for long-tail sitemap ingest. Increase before raising ingest concurrency. |

Related quick references: `.env.example` has the editable template values,
`apps/api/README.md` summarizes rate-limit header behavior, and
`apps/scheduler/src/infra/farvater_http.py` documents the Farvater breaker.
