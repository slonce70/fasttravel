# Full Project Audit — 2026-05-23

Four independent agents audited the stack: Reality Checker (does it work?),
Code Reviewer (statics), Security Engineer (attack surface), Database
Optimizer (schema/queries). One human run hit infra + dataset + perf
benchmarks.

This document is the **consolidated picture** + what was fixed during the
audit + what remains.

---

## Headline

* **Reality:** **24/24 PASS, 0 BROKEN.** Every endpoint, every cron job,
  every UI flow that the recent commits claimed to ship actually works
  end-to-end.
* **Statics:** 4 CRITICAL + 5 IMPORTANT + 5 NICE-TO-HAVE issues found.
* **Security:** 2 CRITICAL + 3 HIGH + 4 MEDIUM + 7 LOW; **no remote
  code execution, no SQL injection, no secret leak in git history**.
* **Database:** 3 CRITICAL + 6 IMPORTANT performance/integrity items.
* **Infra:** all 7 containers healthy; 0 errors in 5-min logs; build
  pass; tests run; perf p95=187ms on 20-concurrent search.

**Bottom line:** the system is production-grade-with-caveats. The two
single biggest risks were both addressed during the audit:

1. **MV refresh stampede** — `_refresh_one_hotel` and `refresh_worker`
   were calling `REFRESH MATERIALIZED VIEW` without `CONCURRENTLY`,
   locking every calendar/search read for the duration. Combined with
   an uncapped `refresh:queue`, this was a single-curl DoS.
   **Fixed:** CONCURRENTLY everywhere + queue length cap (200) + per-hotel
   5-min Redis lock already in place.

2. **Live refresh wrote duplicates** — the worker had no 12h dedup,
   unlike snapshot_farvater. Poisoned percentiles, swelled storage.
   **Fixed:** dedup helper inlined in `_persist_prices`. Smoke confirmed
   `fetched=32 inserted=0 all_deduped=32` on a repeat call.

## Remaining issues — by priority

### Should fix before public launch

1. **Security #1 + #3 — Per-IP rate limit on `/api/hotels/*/refresh`.**
   Queue cap (F-5) prevents OOM. But an attacker rotating `hotel_id`
   from a single IP can still fill 200 slots, freeze new refreshes for
   5 min, and burn farvater's per-IP budget on our behalf. Add
   `slowapi` middleware: 10 refreshes/IP/hour. Estimated effort: 30 min.
2. **Security #4 — Rotate Postgres + Grafana defaults.** `.env` ships
   with `fasttravel_dev_change_me` / `admin`. Prod deploy must
   `generate-secret 32` for each. Add a `infra/secrets-bootstrap.sh`.
3. **DB Optimizer #1 — UNIQUE on `price_observations` natural key.**
   PK is `(id, observed_at)` only. Two concurrent writers in the same
   second can both INSERT the same `(hotel_id, operator_id, check_in,
   nights, meal_plan, observed_at)`. Risk today low (single scheduler);
   rises when we add ittour ingest. Migration is non-trivial (need
   `ON CONFLICT DO NOTHING` everywhere too).
4. **Code Reviewer #5 — `snapshot_hot` doesn't check `refresh:hotel:{id}`
   lock.** User refresh sets a 5-min lock; hot-priority ignores it and
   may queue redundant work. Read the lock before LPUSH.
5. **Code Reviewer #7 — `farvater_client` extraction.** `_fetch_calendar`
   is duplicated between `snapshot_farvater` and `refresh_worker`. Both
   point at the same external API. Extract to `apps/ingest/clients/`
   (already exists as scaffolding from earlier).

### Should fix before scaling past ~5× current data

6. **DB Optimizer #4 — Narrow `current_prices` window.** Currently
   `WHERE check_in <= +90 days`; if we widen retention or reduce
   prune cadence the MV scales linearly. Add `observed_at >= NOW() -
   INTERVAL '7 days'` to MV definition (new migration). Move
   `price_baselines` refresh to nightly (data changes slowly).
7. **DB Optimizer #5 — `raw_payload` is 99.88% NULL.** Either populate
   or drop. Current overhead is small (~3.4MB) but the column choice
   confuses future contributors.
8. **DB Optimizer #7 — `ix_price_obs_deal_window (observed_at, check_in)`
   leads with non-selective column.** Replace with `(check_in,
   observed_at)` or drop entirely (calendar idx covers).
9. **Code Reviewer #8 — `redis.keys('hot:hotel:*')` in `snapshot_hot`.**
   O(N) blocking. Switch to `SCAN`. Today set is tiny; rule is "switch
   before 10k keys".
10. **Reality #1 — Container drift on merge.** Image must be rebuilt
    after every `apps/api/` or `apps/scheduler/` commit. Add to CI:
    `docker compose build api scheduler && docker compose up -d`. Or
    enable hot-reload (`uvicorn --reload`, volume-mount src) in dev
    overlay.

### Nice-to-have / future

11. **Code Reviewer IMPORTANT #3** — PaxPicker dialog has no focus trap.
12. **Security MEDIUM #5** — Validate `photo_url` starts with `https://`
    at scrape time (defence-in-depth; React text escape covers today).
13. **Security MEDIUM #9** — Validate hotel exists before INCR
    `hot:hotel:{id}` (memory leak on iterated IDs; bounded by TTL).
14. **Security LOW** — Add `before_send` Sentry scrubber for DATABASE_URL.
15. **DB MINOR #10** — Drop unused `hotels.coords POINT` column.
16. **DB MINOR #14** — Add weekly `pg_dump` cron to R2.
17. **Code Reviewer NICE-TO-HAVE #1** — Detect deals: read `source` from
    `raw_payload->>'source'` instead of `deep_link LIKE` so future

## What's already done well (praise across all 4 audits)

* `.env` correctly gitignored and not in history (`git log --diff-filter=A`
  confirms).
* **Zero SQL injection** vectors — all `text()` uses named bindparams; the
  only f-string in SQL (`_price_expr` in `search_service.py`) reads from
  a hardcoded whitelist (`_NIGHTS_COLUMN`).
* `escape_markdown_v2` applied to every DB-derived string in
  `post_deals._render_deal`.
* React auto-escapes — no `dangerouslySetInnerHTML` anywhere.
* Sentry `send_default_pii=False`.
* Affiliate `<a>` always carries `rel="nofollow sponsored noopener"`.
* nginx has HSTS, X-Content-Type-Options, X-Frame-Options, rate-limit
  100 req/min/IP on `/api/*`.
* Postgres + Redis ports removed in prod compose overlay
  (`docker-compose.prod.yml`).
* Grafana behind nginx basic-auth, `GF_USERS_ALLOW_SIGN_UP=false`.
* `meal_normalizer` is a single source of truth, well-documented, passes
  unknown codes through cleanly.
* `refresh_views.py` has the correct CONCURRENTLY fallback with a
  specific `ObjectNotInPrerequisiteStateError` catch.
* `refresh_worker_loop` is cancelled cleanly before `scheduler.shutdown()`.
* `post_deals` per-deal try/except keeps one failed Telegram send from
  poisoning the batch.
* pg_partman 5.x integration solid — weekly partitions inherit indexes,
  `cleanup_partitions.py` calls `partman.run_maintenance` daily.
* All 3 MVs have UNIQUE indexes matching their GROUP BY (verified by
  running `REFRESH CONCURRENTLY` on all three).
* Migrations 001-006 are reversible.
* Autovacuum healthy (`n_dead_tup ≤ 23` everywhere).
* `log_min_duration_statement=500` is on — slow query log live.
