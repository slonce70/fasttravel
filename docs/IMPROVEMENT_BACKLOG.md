## Executive summary

Synthesis of 61 verified findings into a prioritized backlog. After deduplication (7 merge clusters: HotelCard dead group-hover x2, Skeleton role=status x2, carousel active-thumbnail broad+narrow, brand-200 undefined ring x2, "Преміум 120+" no-op x2, text_during_wizard dead-branch subsumed by stale-FSM bug, CalendarSkeleton responsive mismatch x2) and one governance split (text-slate-400 DealCard instances peeled into owner-decisions), 50 distinct items remain across five lanes. There are no P0 data-loss or security items. The single highest-impact correctness bug is the bot's "Преміум 120+" budget button, a silent no-op identical to "Без обмежень". The most user-visible UX wins are the missing search loading-state, the amber-on-white CTA contrast failures (WCAG 2.15:1), the bot when-step exact-day query (window labels collapse to a single day), and the Stars Ukrainian-pluralization defect. Owner-gated items (deal/promo rendering, notification cadence, launch decisions) are isolated in the owner-decisions lane regardless of their underlying category.

## Maturity note

This is a mature, production-grade codebase well past a closed audit (~130/134 findings fixed), passing lint+mypy+tests across all five services. The backlog reflects that: it is overwhelmingly polish, accessibility refinement (color-only state, missing aria-live, contrast tokens, reduced-motion, live-region spam), and test-coverage gaps on genuinely logic-heavy files (OffersList branches, HotelView cooldown gate, the GDPR profile delete flow), plus a handful of real but low-blast-radius bugs. There is NO P0 data-loss, security, or outage item in the entire set. The most consequential real bugs are user-facing-misleading-control issues in the Telegram bot (Преміум no-op, when-step exact-day, stale FSM on menu-tap) and a pagination-determinism bug in /api/deals — all bounded, none catastrophic. Many "bugs" are reachable only via hand-edited/stale URLs (показуємо 1001-50, nights out-of-range, NaN metadata) or are slow/narrow resource issues (ThrottleMiddleware dict growth, snapshot coroutine warning). Do not inflate: this is a finished MVP receiving careful final-mile hardening, not a project with structural debt.

### [P1/M] Search results grid has no loading state during server navigation
- **Surface:** web:search
- **Files:** apps/web/src/app/search/page.tsx:26, apps/web/src/components/SearchForm.tsx:126, apps/web/src/components/SearchSortControl.tsx:33
- **What:** The search page is force-dynamic, so every submit/pagination/sort change is a full server round-trip re-running searchHotels(), with zero feedback: results stay frozen, the submit button never enters a pending state (no useTransition/isPending/disabled/aria-busy anywhere), there is no skeleton, and no loading.tsx exists. On the $0 cold-start backend this round-trip can take seconds, making the form feel broken and inviting double-submits.
- **Fix:** Lead with useTransition so the submit button (SearchForm) and sort control (SearchSortControl) show a pending/disabled state during the in-page fetch (avoids the form-remount/focus-loss pitfall of a route loading.tsx). Secondarily add apps/web/src/app/search/loading.tsx (heading shell + ~6 skeleton HotelCards reusing ui/Skeleton) for fresh navigations into /search.
